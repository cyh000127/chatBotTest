# Support Handoff Chatbot Relay

## 1. 목적

이 문서는 지원 이관이 어떤 대화 표면에서 이루어지는지와 구현 순서를 고정한다.

핵심 해석은 다음과 같다.

- 지원 이관은 새 채팅방을 만들지 않는다.
- 지원 이관은 실시간 상담방 연결을 전제로 하지 않는다.
- 지원 이관은 기존 챗봇 대화창 안에서 운영자 후속 답변을 중계하는 흐름이다.
- 사용자는 계속 같은 챗봇 대화창에서 안내와 후속 답변을 받는다.
- 운영자는 별도 운영 표면에서 이관 항목을 확인하고 답변 또는 처리 결과를 남긴다.
- 봇은 운영자 답변과 처리 상태를 사용자 대화창으로 전달한다.

## 2. 이관 조건

아래 경우에는 모델보다 지원 이관이 우선한다.

- 반복 실패가 한도에 도달한 경우
- 모델이 현재 입력을 회수하지 못한 경우
- 문제가 발생했다고 사용자가 명시한 경우
- 상담원 또는 관리자 연결을 요청한 경우
- 민감하거나 공격적인 입력으로 자동 대응이 부적절한 경우
- 입력 후보가 충돌하거나 수동 확정이 필요한 경우
- 운영자가 직접 관리해야 하는 상태로 분류된 경우

## 3. 이관 의미

`support.escalate`는 사용자에게 즉시 운영자가 연결된다는 뜻이 아니다.

`support.escalate`는 아래 상태를 만드는 정책 신호다.

- 현재 세션을 지원 이관 상태로 표시한다.
- 사용자에게 운영 검토 접수 안내를 보여준다.
- 운영 표면에서 확인할 수 있는 이관 항목을 생성할 준비를 한다.
- 이관 사유, 최근 대화 요약, 현재 단계, 실패 횟수, 사용자 입력을 함께 남긴다.
- 운영자 답변이 생기면 기존 챗봇 대화창으로 전달할 수 있게 한다.

## 4. 사용자 경험 원칙

사용자에게는 아래 기준으로 안내한다.

- 실시간 상담처럼 약속하지 않는다.
- 새 대화창으로 이동하라고 안내하지 않는다.
- 같은 챗봇 대화창에서 후속 안내를 받을 수 있다고 설명한다.
- 사용자가 계속 메시지를 보내면 지원 이관 상태의 추가 사용자 메시지로 기록한다.
- 운영자 답변이 도착하면 봇이 대화창에 이어서 전달한다.
- 운영자 처리 전에는 자동 저장이나 임의 확정을 하지 않는다.

권장 안내 문구:

> 요청을 운영 검토로 넘겼습니다. 새 채팅방을 만들지 않고 이 대화창에서 후속 안내를 받을 수 있습니다.

## 5. 런타임 상태

지원 이관 상태는 최소한 아래 정보를 가져야 한다.

- 이관 식별자
- 사용자 식별자
- 세션 식별자
- 이관 사유
- 현재 단계
- 최근 대화 요약
- 실패 횟수
- 사용자 원문
- 운영자 답변 대기 여부
- 마지막 운영자 답변 시각
- 종료 여부

## 6. 구현 계획

### 6.1 문서 고정

- 지원 이관이 실시간 상담방 연결이 아님을 문서에 명시한다.
- 같은 챗봇 대화창 안에서 운영자 후속 답변을 중계한다는 원칙을 고정한다.
- 사용자 안내 문구와 정책 어휘를 맞춘다.

### 6.2 이관 상태 계약 추가

- `support.escalate`가 발생할 때 세션에 지원 이관 상태를 기록한다.
- 이관 사유와 최근 대화 요약을 구조화한다.
- 반복 실패, 사용자 요청, 수동 해결 필요를 동일한 이관 계약으로 정렬한다.

### 6.3 사용자 메시지 수집

- 지원 이관 상태에서 사용자가 추가 메시지를 보내면 새 명령으로 오인하지 않는다.
- 추가 메시지는 이관 항목의 대화 이력으로 누적한다.
- 명시적인 재시작 명령은 기존처럼 현재 흐름을 재시작한다.

### 6.4 운영자 답변 중계

- 운영자가 답변을 남길 수 있는 내부 입력 계약을 만든다.
- 운영자 답변은 봇 메시지로 사용자에게 전달한다.
- 전달 실패 시 재시도 또는 운영 실패 상태를 남긴다.

### 6.5 종료와 재개

- 운영자가 해결 완료, 응답 대기, 조치 불가 같은 상태를 남길 수 있게 한다.
- 해결 완료 후 사용자는 일반 챗봇 흐름으로 돌아간다.
- 미해결 상태에서는 사용자의 추가 메시지를 기존 이관 항목에 이어 붙인다.

### 6.6 검증

- 반복 실패 후 이관 상태가 생성되는지 검증한다.
- 상담원 요청이 이관 상태로 연결되는지 검증한다.
- 이관 상태의 추가 사용자 메시지가 대화 이력으로 누적되는지 검증한다.
- 운영자 답변이 같은 챗봇 대화창으로 전달되는지 검증한다.
- 재시작 명령이 이관 상태에서도 우선 처리되는지 검증한다.

## 7. 제외 범위

아래 항목은 현재 지원 이관의 필수 구현 범위가 아니다.

- 별도 상담방 생성
- 외부 상담 도구 연결
- 항상 즉시 응답하는 실시간 상담 보장
- 모델이 운영자처럼 장문 상담을 대신 수행하는 기능
- 운영자 승인 없이 사용자의 값을 확정 저장하는 기능

## 8. 최종 기준

지원 이관의 최종 기준은 다음과 같다.

- 사용자는 기존 챗봇 대화창을 유지한다.
- 운영자는 별도 운영 표면에서 확인하고 답변한다.
- 봇은 운영자 답변을 사용자 대화창으로 중계한다.
- 실시간 연결이 아니라 운영 후속 처리 흐름으로 안내한다.
- 모든 자동 처리 실패는 저장 확정이 아니라 이관 또는 재질문으로 끝난다.

# 자유형식 일괄 입력 가이드

이 문서는 `OCR 없이`, `LLM 없이`, `규칙 기반 + 문맥 기반`으로 자유형식 입력을 처리하는 방법을 공부용으로 정리한 문서다.

대상 예시는 아래 3개 필드다.

- `이름`: 입력 필드
- `생년월일`: 버튼형 또는 단계형 입력
- `거주지`: 입력 필드

핵심 목표는 이거다.

- 사용자가 한 번에 길게 입력해도 최대한 구조화한다.
- `시`, `구`가 애매하면 내부 정규화 규칙으로 먼저 맞춘다.
- 그래도 못 맞추면 봇이 다시 물어본다.
- 최종 저장은 항상 사용자 확인 후 한다.

---

## 1. 가장 현실적인 구현 방향

이 케이스는 `AI 파서`보다 아래 조합이 더 만들기 쉽고 안정적이다.

1. `최근 문맥 유지`
2. `간단한 규칙 기반 추출`
3. `시/구 정규화 테이블`
4. `못 찾으면 다시 묻기`

즉, 자유문장을 완벽히 이해하려고 하지 말고:

- `이름 후보`
- `거주지 후보`
- `시/구 후보`

정도만 먼저 뽑고,
부족하면 follow-up 질문으로 채우는 방식으로 간다.

## 2. 추천 UX

### 2.1 기본 입력 방식

- `이름`: 자유 입력
- `생년월일`: 버튼형 또는 단계형
  - 연도 선택
  - 월 선택
  - 일 선택
- `거주지`: 자유 입력

### 2.2 자유형식 일괄 입력 허용

사용자가 아래처럼 한 번에 보낼 수 있다.

```text
이름 김민수, 생일은 1998년 4월 20일이고 서울 강남 살아요
```

또는

```text
김민수 / 서울 강남구 / 1998-04-20
```

또는

```text
민수, 서울 강남, 98.04.20
```

이 경우 시스템은 먼저 초안을 만든다.

```json
{
  "name": "김민수",
  "birth_date": "1998-04-20",
  "residence_raw": "서울 강남",
  "city_candidate": "서울특별시",
  "district_candidate": "강남구",
  "confidence": {
    "name": "high",
    "birth_date": "high",
    "residence": "medium"
  }
}
```

그다음 사용자에게 확인한다.

```text
아래 정보로 저장할까요?
- 이름: 김민수
- 생년월일: 1998-04-20
- 거주지: 서울특별시 강남구
```

## 3. 처리 원칙

### 3.1 무조건 규칙 우선

아래는 AI 없이 먼저 처리한다.

- 이름 분리
- 날짜 패턴 인식
- 시/도 키워드 매칭
- 구/군/시 키워드 매칭
- 별칭 -> 정식 행정구역명 매핑

### 3.2 애매하면 바로 다시 묻기

예:

- `서울 강남`
- `수원 영통`
- `성남 분당`
- `고양 일산`

이건 사람이 보기엔 이해되지만,
시스템은 내부적으로 정식 명칭으로 바꿔야 한다.

예:

- `서울 강남` -> `서울특별시 강남구`
- `수원 영통` -> `경기도 수원시 영통구`
- `성남 분당` -> `경기도 성남시 분당구`
- `고양 일산` -> `경기도 고양시 일산동구` 또는 `일산서구`

여기서 후보가 1개면 자동 정규화,
후보가 여러 개면 follow-up을 한다.

## 4. 추천 처리 플로우

### 4.1 1차 파싱

입력:

```text
저는 김민수고 서울 강남 살아요. 생일은 98년 4월 20일이에요
```

1차 처리:

1. 이름 추출
2. 날짜 추출
3. 거주지 raw text 추출
4. `서울`, `강남` 토큰 분리
5. 시/구 정규화 테이블 조회

결과:

- 이름 = `김민수`
- 생년월일 = `1998-04-20`
- 거주지 raw = `서울 강남`
- city = `서울특별시`
- district = `강남구`

### 4.2 확인 단계

```text
확인했어요.
- 이름: 김민수
- 생년월일: 1998-04-20
- 거주지: 서울특별시 강남구

맞으면 [확인], 수정하려면 [수정]을 눌러주세요.
```

### 4.3 후보가 여러 개면 follow-up

입력:

```text
저는 이수민이고 고양 일산 살아요
```

문제:

- `고양`
- `일산`

만으로는 `일산동구`, `일산서구`를 확정하기 어렵다.

follow-up:

```text
거주지를 조금 더 확인할게요.
고양시에서는 아래 중 어디에 해당하나요?
1. 일산동구
2. 일산서구
```

## 5. 시/구 정규화 규칙

이 부분이 핵심이다.

자유입력은 내부적으로 아래 순서로 처리한다.

1. 원문 정리
2. 별칭 치환
3. 시/도 후보 찾기
4. 구/군/시 후보 찾기
5. 후보 수 판단
6. 1개면 확정
7. 2개 이상이면 follow-up

### 5.1 전처리 규칙

- 공백 normalize
- 쉼표, 슬래시, 하이픈 제거 또는 구분자로 통일
- `서울시`, `서울 특별시`, `서울` -> `서울`
- `경기`, `경기도` -> `경기`
- `부산시`, `부산` -> `부산`
- `광역시`, `특별시`, `특별자치시`, `특별자치도`는 내부 표준명으로 변환

예:

```text
"서울시 강남"
-> "서울 강남"

"경기 수원 영통"
-> "경기 수원 영통"
```

| 사용자 입력 | 1차 해석 | 처리 |
| --- | --- | --- |
| 서울 강남 | 서울특별시 강남구 | 바로 확정 가능 |
| 경기 성남 분당 | 경기도 성남시 분당구 | 바로 확정 가능 |

| 중구 | 여러 지역에 존재 | 시/도 먼저 다시 물어야 함 |

## 7. follow-up 규칙

이 부분은 실제 봇 UX에 바로 들어갈 수 있게 만든 규칙이다.

### 7.1 시/도를 못 찾은 경우

입력:

```text
강남 살아요
```

응답:

```text
거주지를 조금 더 확인할게요.
어느 시/도에 해당하나요?
예: 서울특별시, 경기도
```

### 7.2 시/도는 찾았는데 구를 못 찾은 경우

입력:

```text
서울 살아요
```

응답:

```text
서울특별시까지 확인했어요.
어느 구에 거주하시나요?
예: 강남구, 서초구, 송파구
```

### 7.3 후보가 여러 개인 경우

입력:

```text
고양 일산 살아요
```

응답:

```text
일산은 두 곳으로 나뉘어요.
아래 중 어디에 해당하나요?
1. 일산동구
2. 일산서구
```

### 7.4 사용자가 비표준 표현을 쓴 경우

입력:

```text
서울 강남쪽이요
```

응답:

```text
거주지를 `서울특별시 강남구`로 이해했어요.
맞으면 [확인], 아니면 [수정]을 눌러주세요.
```

### 7.5 여러 번 실패한 경우

응답:

```text
거주지를 정확히 확인하지 못했어요.
시/도와 구를 나눠서 다시 입력해주세요.
예: 서울특별시 강남구
```

필요하면 여기서 버튼형으로 강제 전환한다.

## 8. 추천 저장 구조

처음부터 복잡하게 만들 필요는 없고, 아래 정도면 충분하다.

### 8.1 profile_drafts

자유입력으로 뽑은 초안을 저장한다.

| 컬럼 | 설명 |
| --- | --- |
| `id` | draft id |
| `user_id` | 사용자 id |
| `raw_text` | 사용자가 보낸 원문 |
| `name_candidate` | 이름 후보 |
| `birth_date_candidate` | 생년월일 후보 |
| `residence_raw` | 원본 거주지 문자열 |
| `city_candidate` | 시/도 후보 |
| `district_candidate` | 구/군/시 후보 |
| `parse_status` | `parsed`, `ambiguous`, `needs_followup`, `confirmed` |
| `created_at` | 생성 시각 |

### 8.2 conversation_sessions

현재 어떤 질문을 하고 있는지 저장한다.

| 컬럼 | 설명 |
| --- | --- |
| `id` | session id |
| `user_id` | 사용자 id |
| `current_intent` | 예: `profile_create`, `profile_edit` |
| `pending_slot` | 예: `city`, `district`, `birth_day` |
| `draft_id` | profile draft 참조 |
| `last_bot_question` | 마지막 질문 |
| `retry_count` | 재질문 횟수 |
| `updated_at` | 갱신 시각 |

### 8.3 admin_area_aliases

시/구 정규화 테이블이다.

| 컬럼 | 설명 |
| --- | --- |
| `id` | row id |
| `level` | `city`, `district` |
| `canonical_name` | 정식 명칭 |
| `alias` | 별칭 |
| `parent_city` | district면 상위 시/도 |
| `is_ambiguous` | 애매한 별칭 여부 |

## 10. 추천 구현 순서

### Step 1

아래만 먼저 구현한다.

- 이름 추출
- 날짜 추출
- 거주지 raw 저장
- 시/도 alias 매핑
- 구/군 alias 매핑
- 확인 메시지

### Step 2

그다음 추가한다.

- 애매한 시/구 follow-up
- 세션 저장
- retry_count
- 버튼형 확인/수정

### Step 3

마지막에 추가한다.

- 최근 문맥 기반 추정
- 여러 입력 포맷 지원 강화
- 다국어 표현 대응

## 11. 구현시 주의점

- 자유입력은 `바로 저장`하지 말고 `초안 -> 확인 -> 저장`으로 가야 한다.
- 거주지는 특히 정식 행정구역명으로 정규화한 뒤 저장해야 한다.
- 애매한 별칭은 억지로 확정하지 말고 다시 물어야 한다.
- `생년월일`은 가능하면 버튼형/단계형으로 받는 편이 더 안전하다.
- 사용자가 한 번에 보내도, 내부적으로는 여러 슬롯으로 나눠 저장해야 한다.

## 12. 한 줄 요약

이 문제는 AI 없이도 충분히 풀 수 있다.

정답은:

- `자유입력 초안 생성`
- `시/구 정규화 테이블`
- `애매하면 follow-up`
- `최종 확인 후 저장`

이 4단계다.
